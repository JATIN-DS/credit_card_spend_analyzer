Get your Google OAuth Client ID
This lets the app read your Gmail statement PDFs. It's free, no credit card required, and everything stays in your browser.
  1. Open the Google Cloud Console at console.cloud.google.com and sign in.
  2. Create a project: click the project dropdown (top bar) → New Project → name it (e.g. "Statement Analyzer") → Create, then select it.
  3. Enable the Gmail API: left menu → APIs & Services → Library → search "Gmail API" → open it → Enable.
  4. Configure the OAuth consent screen: APIs & Services → OAuth consent screen → choose External → Create. Fill App name, User support email, and Developer contact email → Save and Continue. On the Scopes step, just Save and Continue. On Test users, click Add Users and add your own Gmail address (and any others who will use the app) → Save. (The app stays in "Testing" mode; only listed test users can connect.)
  5. Create the credential: APIs & Services → Credentials → Create Credentials → OAuth client ID → Application type: Web application → give it a name. Under "Authorized JavaScript origins" click Add URI and enter the EXACT origin where you open this app (for local use, e.g. http://localhost:8000) → Create.
  6. Copy the Client ID (it looks like 1234567890-abc.apps.googleusercontent.com).
  7. Go to the Settings tab → paste it into "Step 1 · Google OAuth Client ID" → Save settings → click "Step 2 · Connect Gmail" and grant read-only Gmail access.
Tips
  • The Authorized JavaScript origin must match your URL exactly (scheme + host + port). If you open the file via file://, serve it instead with python3 -m http.server 8000 and add http://localhost:8000.
  • If you see "Access blocked / app isn't verified", make sure your email is added as a Test user (or publish the consent screen).
  • Your Client ID, tokens and PDF passwords are stored only in your browser's localStorage — nothing is sent anywhere except Google.
Add a new bank's extraction logic (e.g. SBI)
Statements are parsed entirely in your browser (Python via Pyodide) by per-bank parsers. Banks without a custom parser fall back to a best-effort generic parser, which may misread totals/transactions — so add a dedicated parser for full accuracy.
  1. In Settings, add the card with the correct Bank selected, then Fetch once. Open your browser's DevTools Console — the raw extracted text of each statement is printed there (look for a block labelled RAW EXTRACTED TEXT). Copy that text for one sample statement.
  2. Open this project in an AI coding editor (e.g. Cursor) and ask it, pasting the raw text: "Add an extraction parser for <BANK> credit card statements in py/parsers.py. Here is the raw extracted statement text: <paste>. Follow the same normalized output shape as parse_icici / parse_hdfc, and register it in _REGISTRY."
  3. The only file to modify is py/parsers.py. The agent should implement a function parse_<bank>(extracted, card_label) that returns the normalized shape: { "card", "bank", "transactions": [{ "date" (YYYY-MM-DD), "merchant", "amount", "type" ("debit"|"credit"), "card", "category" }], "summary": { "card", "bank", "totalDue", "minDue", "dueDate", "statementPeriod" }, "error" } and add an entry to the _REGISTRY dict (keyed by the uppercase bank code, e.g. "SBI") so parse_statement dispatches to it.
  4. Reload the app. Dispatch is by the card's Bank field, so the new bank automatically uses the new parser — no other files need changing.
The existing parse_hdfc and parse_icici in py/parsers.py are good reference implementations to point the agent at.
